## 5. Operating Taxonomy

The durable work vocabulary is:

- Project Goal: the persistent top-level objective authorized by this document.
- Mission: one terminal-bounded scientific campaign under the Project Goal.
- Initiative: one bounded coherent objective with explicit done conditions.
- Portfolio: the current set of open questions, alternatives, and lineages.
- Study: one causal question or materially distinct structural axis.
- Batch: a frozen adaptive set of counted trials within one Study.
- Job: one bounded computation, data, runtime, or materialization execution.
- Executable: one immutable component composition, parameters, and bound
  scientific/runtime context.
- Lineage: a parent-child research relationship among related Executables.
- Decision: preserve, prune, deepen, rotate, contrast, recombine, or synthesize.
- Repair: engineering work that preserves the scientific identity it repairs.
- Release: a frozen ONNX, EA, evidence, and handoff bundle.

- [MUST] OD-TAX-001 These concepts have separate identities and lifecycle rules.
- [MUST] OD-TAX-002 A single global maturity cursor is not used. Each immutable
  Executable carries its own evidence depth.
- [MUST] OD-TAX-003 Evidence depth is described as discovery, confirmation,
  execution_proof, materialization, or release.
- [MUST] OD-TAX-004 Initiative close does not automatically close the Mission.
- [MUST] OD-TAX-005 A Study or Batch does not become new merely through renaming.
- [MUST] OD-TAX-006 Temporal work history does not become a Python dependency tree.
- [MUST] OD-TAX-007 Work records use stable identities without generation prefixes.
- [MUST] OD-TAX-008 The final active folder layout follows stable responsibilities,
  not copied folders for each Initiative or evidence depth.
- [MUST] OD-TAX-009 Trials, claims, and negative memory bind to immutable
  Executables. A Lineage never merges distinct Executable evidence.

## 11. External Symbol And External Data Eligibility

External symbols are considered actively without prior preference. They are
used when they add positive information value and are executable in the actual
US100 runtime.

An external source eligibility record has exactly one state:

- context_only
- historical_audited
- runtime_eligible
- suspended

Candidate freeze is a separate binding between a frozen Executable and an exact
source, mapping, eligibility receipt, and runtime identity.

State permissions are:

- context_only: qualitative context and feasibility only; no performance,
  scheduler, evidence, candidate, or negative-memory use.
- historical_audited: historical mechanics and eligibility audit only; no
  performance Batch or scientific priority.
- runtime_eligible: SourcePermit may be issued for a preregistered Batch.
- suspended: no new SourcePermit; dependent sleeves fail closed; only diagnosis
  and recertification are allowed.

Historical audit may move context_only to historical_audited. Runtime
availability proof may move historical_audited to runtime_eligible. Drift moves
runtime_eligible to suspended. Same-semantics recertification may restore
runtime_eligible; semantic change requires a new source contract.
